# 分布式社区调度方案（待开发设想）

> 读者：Claude Opus 4.6
> 目的：明确当前方案、目标方案、迁移路径与技术关键点，确保后续开发方向一致。

## 一句话摘要

将 Panel 升级为 **社区创建与调度控制平面**，多个 Daemon 组成 **可选节点池** 为同一社区服务；短期采用 **共享 DB + Redis PubSub（方案 A）** 实现“多入口分布式”，中期按社区分库扩展吞吐。

---

## 1. 原有方案（现状）

### 架构
- **Panel**：只做多 Daemon 管理与监控
- **Daemon**：单社区服务核心（WebSocket/频道/消息/媒体）
- **数据库**：单一 PostgreSQL，Daemon 直接读写

### 现状特点
- 社区创建不在 Panel
- Daemon 仍是单社区模型
- 对同一社区仅有单节点入口
- 负载无法在多个节点分散

---

## 2. 目标方案（你的设想）

### 目标能力
- Panel 统一 **创建社区**
- 一个社区可绑定多个 Daemon 节点（节点池）
- 客户端可连接任意节点
- 节点间 **异步同步数据 + 状态共享**
- 通过调度降低单节点压力，实现“联邦式多入口服务”

### 关键定义
- **控制平面（Panel）**：社区创建/调度/节点选择
- **数据平面（Daemon Pool）**：消息处理、连接与推送
- **共享存储**：阶段 1 用共享 DB + Redis PubSub 保持一致性

---

## 3. 推荐实施路径

### 阶段 1（方案 A）
**共享 PostgreSQL + Redis PubSub**

#### 目标
- 多 Daemon 作为同一社区的“入口节点”
- 所有节点写同一 DB，消息入库后通过 Redis 广播给其他节点

#### 优点
- 工程成本低
- 顺序性由 DB 保证
- 可快速验证“多入口 + 调度”

#### 代价
- DB 为中心点
- 高并发社区仍可能成为瓶颈

---

### 阶段 2（扩展）
**按社区分库/分片 + 读写分离**

#### 目标
- 热社区独立数据库
- Panel 调度时分配社区到具体 DB shard
- 每个 Daemon 按社区路由到对应 DB

#### 效果
- 水平扩展数据库吞吐
- 降低单库压力

---

## 4. 关键改动点（实现层面）

### 4.1 数据模型（必须）

新增关系：
- `communities`（已存在）
- `community_daemon_nodes`（新增）

建议字段：
```
community_id
daemon_id
weight
role (primary/edge)
enabled
```

### 4.2 Panel 负责社区创建
- Panel UI + API：创建社区 → 写入 `communities`
- Panel 管理社区绑定的 Daemon 节点池

### 4.3 Daemon 变为“多社区入口节点”
- Daemon 接收 `community_id` 作为上下文
- WebSocket/REST 路由需携带社区信息
- 频道/成员/消息表按 `community_id` 分区（或逻辑隔离）

### 4.4 负载调度策略（可逐步演进）

#### 最小可用（Phase 1）
- Panel 返回节点列表（按 health/weight）
- 客户端挑选最佳节点连接

#### 改进（Phase 1.5）
- 加入节点负载指标（CPU/MEM/WS 连接数）
- Panel 返回排序后的 TopN

### 4.5 实时同步

方案 A 中：
- 消息入库后 Redis PubSub 广播
- 各 Daemon 监听频道事件并推送在线用户
- 离线消息仍在 DB 中统一处理

---

## 5. 预期效果

- **负载均衡**：用户可分布在不同 Daemon 入口
- **容灾能力**：单节点故障不影响整个社区
- **扩展性**：后续可通过分库分片突破单 DB 限制
- **体验提升**：靠近用户的节点能提供更低延迟

---

## 6. 关键技术难点与风险

| 领域 | 难点 | 风险 |
|------|------|------|
| 一致性 | 多入口写入顺序 | 乱序/重复消息 |
| 状态同步 | 在线状态/typing/read | 跨节点延迟 |
| DB 压力 | 单 DB 大流量 | 性能瓶颈 |
| 媒体存储 | 多节点访问 | 需对象存储统一入口 |

> 方案 A 通过“共享 DB + Redis”规避多活一致性复杂度。

---

## 7. Claude Opus 4.6 可直接执行的工作清单

1. **数据库**：新增 `community_daemon_nodes` 表
2. **Panel API**：
   - 创建社区（写 communities）
   - 绑定/解绑 Daemon 节点
   - 获取社区节点池（供客户端调用）
3. **Daemon 接入**：
   - 新增 `community_id` 上下文
   - 多社区路由支持
4. **客户端逻辑**：
   - 拉取节点池并选择连接
   - 断线自动切换备用节点
5. **Redis PubSub**：
   - 消息写入后广播
   - 其他节点监听并转发
