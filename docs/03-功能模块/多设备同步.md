# 多设备同步

## 模块概述

多设备同步模块确保同一用户在不同设备上看到一致的消息状态。通过同步游标（Sync Cursor）机制追踪每个设备在每个频道的读取进度，支持增量同步，避免重复拉取已读消息。

## 核心概念

### 同步游标

同步游标记录每个设备在每个频道中最后读取到的消息 ID：

- 三元组 `(member_id, device_id, channel_id)` 唯一标识一个游标
- 游标值是最后读取的消息的 UUID
- 客户端每次同步后更新游标

### 增量同步

基于游标的增量同步流程：

1. 客户端提交当前游标（最后读取的消息 ID）
2. 服务端查询该消息之后的所有新消息
3. 返回增量消息列表
4. 客户端处理完成后更新游标

### 设备间消息同步

当用户在设备 A 发送消息后：

- 消息通过 WebSocket 实时推送到在线的设备 B
- 如果设备 B 离线，下次上线时通过增量同步获取新消息
- 每个设备独立维护游标，互不影响

### 冲突解决

由于每条消息有服务端分配的 `server_timestamp` 和 `seq_num`，冲突解决策略简单明确：

- **消息排序**：以 `server_timestamp` 为准，服务器权威排序
- **游标更新**：使用 `UPSERT`（`ON CONFLICT DO UPDATE`），最新的游标覆盖旧值
- **已读状态**：各设备独立追踪，不做跨设备已读同步（由客户端决策）

## 数据模型

### sync_cursors 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `member_id` | UUID | 用户 ID（联合主键） |
| `device_id` | UUID | 设备 ID（联合主键） |
| `channel_id` | UUID | 频道 ID（联合主键） |
| `last_message_id` | UUID | 最后读取的消息 ID |
| `updated_at` | TIMESTAMPTZ | 游标更新时间 |

主键：`(member_id, device_id, channel_id)`

### 关联表

- `messages` 表：通过 `channel_id` 和 `server_timestamp` 查询增量消息
- `read_states` 表：用户级别（非设备级别）的已读状态

## API 端点

同步功能主要通过 WebSocket 协议和内部服务调用实现：

### WebSocket 同步事件

```json
// 客户端请求同步
{
  "type": "SYNC_REQUEST",
  "channel_id": "ch-uuid",
  "last_message_id": "msg-uuid-last-read"
}

// 服务端返回增量消息
{
  "type": "SYNC_RESPONSE",
  "channel_id": "ch-uuid",
  "messages": [...],
  "has_more": false
}
```

### REST API 辅助

消息历史查询端点也支持同步场景：

```
GET /api/v1/channels/{id}/messages?before={cursor}&limit=50
```

## 配置

```toml
[sync]
# 单次增量同步最大消息数
max_sync_batch_size = 200
# 游标过期时间（超过此时间未更新的游标会被清理）
cursor_ttl = "90d"
```

## 示例

### 同步流程

```
设备 A                       Server                     设备 B
  |                            |                           |
  |--- 发送消息 M1 ---------> |                           |
  |                            |--- WebSocket 推送 M1 --> |
  |                            |                           |
  |--- 发送消息 M2 ---------> |                     设备 B 离线
  |                            | [存储 M2]                 X
  |                            |                           |
  |--- 发送消息 M3 ---------> |                           |
  |                            | [存储 M3]                 |
  |                            |                           |
  ...                          |                     设备 B 上线
  |                            |                           |
  |                            | <-- SYNC_REQUEST --------|
  |                            |    (last_message_id=M1)   |
  |                            |                           |
  |                            | --- SYNC_RESPONSE -----> |
  |                            |    (messages=[M2, M3])    |
  |                            |                           |
  |                            | <-- 更新游标 (M3) -------|
```

### 获取未读计数

```rust
// SyncCursorService 内部调用示例
let unread = sync_service
    .get_unread_count(member_id, device_id, channel_id)
    .await?;
// 返回该设备在该频道的未读消息数量
```

### 更新游标

```rust
// 设备处理完消息后更新游标
sync_service
    .update_cursor(member_id, device_id, channel_id, last_message_id)
    .await?;
```

游标更新使用 `UPSERT` 语义，首次记录时 INSERT，后续更新时 UPDATE：

```sql
INSERT INTO sync_cursors (member_id, device_id, channel_id, last_message_id, updated_at)
VALUES ($1, $2, $3, $4, now())
ON CONFLICT (member_id, device_id, channel_id)
DO UPDATE SET last_message_id = $4, updated_at = now()
```
